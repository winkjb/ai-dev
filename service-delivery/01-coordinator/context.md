# service-delivery/01-coordinator

## Purpose
Reactive Tier — Agent #1 in `../support-service-delivery-agent-architecture.md` (named "Coordinator" for consistency with `project-management/01-coordinator`, which does the same functional job — reactive status tracking + flagging — against a different data domain). Tracks ticket status/ownership/SLA clocks; assigns to the correct tech by skillset/severity; flags tickets approaching SLA breach; escalates stalled tickets.

## Contents
- `ticket_report.py` — reads the ticket export (`../data/raw`), drops tickets matching any Queue+Source rule in `../data/reference/excluded-ticket-sources.csv` (out-of-scope queues, plus the Audit & Compliance / Business Continuity + Recurring compound rule), and reports on every remaining in-scope ticket. Assigns a single `Health` label per ticket — Critical Unassigned > Stalled Intake > Stale > Waiting External > Unassigned > Active, worst-condition-wins, used for sorting and the summary counts — plus an `All Flags` column listing every condition that's actually true for that ticket (e.g. `Stale, Waiting External`), and the underlying boolean flag columns for full detail. Also labels every in-scope ticket with a `Ticket Origin` (Human-Generated / System-Generated / Unclassified) via `../data/reference/source-classification.csv` — these tickets stay in the report, this just flags which ones came from an automated tool so they still get reviewed. Run with `python ticket_report.py` from this folder.
- `output/` — generated reports (regenerable; not hand-edited). `ticket-dispatch-report.csv` (every in-scope ticket with Health + flag detail) + `ticket-dispatch-summary.md` (counts, by-queue breakdown).

## 60/30/10 Breakdown
(see `../../principles.md` for the full principle)

- **Programmatic (60%)** — all of it so far. Critical/Stalled-Intake/Stale/Waiting/Unassigned are objective status/queue/age comparisons against constants (`STALE_DAYS`, `WAITING_STATUSES`, `INTAKE_QUEUE`, `DISPATCHED_STATUS`); Health prioritization, sorting, and report formatting are all deterministic.
- **Data-Driven (30%)** — reads `../data/raw` (the ticket export, now including a real `Last Activity Time` column as of 2026-07-08), `../data/reference/excluded-ticket-sources.csv` (human-maintained Queue+Source exclusion rules), and `../data/reference/source-classification.csv` (human-maintained Source → Ticket Origin lookup, added 2026-07-08). No inference involved.
- **AI (10%)** — none currently, same reasoning as `project-management/01-coordinator`.

## Notes
See architecture doc for full inputs/outputs/handoff spec. This folder was renamed from `01-ticket-dispatch` to `01-coordinator` (2026-07-08) purely for cross-project naming consistency — **the implementation stays separate and domain-specific**, not shared code. Per the architecture doc's "Where This Overlaps With the PM Architecture" table, Coordinator/Ticket-Dispatch was deliberately classified as "similar pattern, different data/constraints — likely separate instances," unlike the four roles flagged as "strong candidate to share" (Client Communication, Intake/Triage, Risk/Compliance, Resource Allocation). Script/output filenames stayed ticket-specific (`ticket_report.py`, `ticket-dispatch-report.csv`) since that's genuinely what this role's deliverable is about — only the folder/role name changed.

**Important calibration note:** the source export's own `Due` field is NOT a live SLA clock — it's a static ~24 hour first-response timer set once at ticket creation, so a naive `Due < now` check flags the majority of the queue regardless of actual ticket age and isn't meaningful on its own (same category of problem as PM's "Project Status" field always reading Green). `STALE_DAYS = 7` and the waiting/intake constants are set at the top of `ticket_report.py` — tune there if the definitions need to change.

**2026-07-08 revision:** the export gained a real `Last Activity Time` column, so the old `Neglected` flag (a status+ticket-age approximation of "nobody's touched this") was retired in favor of `Stale` (based on the real last-touch timestamp — strictly more accurate). A new `Stalled Intake` flag was also added: tickets sitting in the `New` queue at all, or in `Dispatched` status outside the New queue — a workflow/routing signal, deliberately with no age threshold, distinct from time-based staleness. Caveat worth watching: if `Last Activity Time` can be bumped by automated/system actions without real human engagement, `Stale` could under-flag things the old status-based `Neglected` would have caught — revisit if that turns out to matter in practice.

**2026-07-08 second revision:** `excluded-ticket-sources.csv` moved from a `Type,Value` single-column scheme to a `Queue,Source` AND/OR rule table (see `../data/reference/context.md`) so it can express compound rules like "Queue X or Y, AND Source Z" — the old scheme could only exclude by a single column at a time. Also fixed a bug where `OUTPUT_CSV`/`OUTPUT_SUMMARY` in `ticket_report.py` referenced `ticket-status-*` filenames that didn't match the actual `ticket-dispatch-*` files being written (a rename that had slipped through incompletely) — next run would have silently produced a second, differently-named pair of output files. Separately, the monitoring-tool Source exclusions (Nable, Auvik, RocketCyber, Solarwinds, Monitoring Alert) were removed from the exclusion list — those tickets are back in scope, now labeled `System-Generated` via the new `Ticket Origin` column instead of being dropped, since they still need human review.
